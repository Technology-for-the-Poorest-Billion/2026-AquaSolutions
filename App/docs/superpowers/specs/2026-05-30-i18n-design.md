# Design — Web portal internationalisation (English / Shona / Ndebele)

**Date:** 2026-05-30
**Status:** Approved (brainstorming complete; awaiting implementation plan).
**Scope:** Add a multi-language UI layer to the Gen-1 Flask web portal, with English, Shona, and Ndebele in v1 and a framework where adding any of the remaining 13 of Zimbabwe's official languages is a one-line config edit plus translation files.

## 1. Context

The Gen-1 app's UI (six templates under `App/backend/templates/`) is currently English-only. `Ideation.md` flagged "Adapts to one of Zimbabwe's 16 official languages" as a design goal from day one; the 2026-05-28 medical/government portal spec deliberately deferred internationalisation as out-of-scope (§9, line 310). This design picks it back up.

The audience split is asymmetric:
- The **government dashboard** and **medical portal** are used by officials and clinicians — many will choose English, but the project's framing requires the UI to support local languages on principle.
- The **community-facing leg** today is SMS, which is out of scope for this design (locked in Section 4). When a public-facing patient screen is added later it will inherit this i18n layer.

## 2. Key design decisions

| Decision | Choice | Rationale |
|---|---|---|
| Scope | **Web portal only** | SMS leg has qualitatively different problems (inbound symptom-keyword tables in each language). Defer to its own design. |
| Priority languages | **English, Shona (sn), Ndebele (nd)** | Largest speaker shares in Zimbabwe; covers the realistic government-medical-community span. Adding any of the other 13 is one line in the `LANGUAGES` dict plus translation files. |
| Selection mechanism | **Top-right picker + cookie + DB column** | Manual choice avoids the "Zimbabwean phones ship configured to English" failure mode of pure Accept-Language. Cookie covers anonymous + login page; DB column persists across devices for signed-in users. |
| Translation sourcing | **Google Translate draft, every entry marked `#, fuzzy`** | Fast enough for the 2026-06-01 demo. The fuzzy flag is the first-class "unverified" state; a banner surfaces this honestly to users. Native review is the only real fix and is flagged in §13. |
| Architecture | **Flask-Babel + `language.py` wrapper** | Standard gettext ecosystem (plurals, tooling, .po files translators expect) with project-specific concerns (DB write, allow-list, unverified banner, picker route) localised in one module rather than scattered across views. |
| Fallback | **Silent fallback to English** | Standard gettext behaviour. Dev mode logs the missing key so we can find gaps. |
| Coverage | **All six templates + flash messages + estimator rationale strings** | These are user-facing. Server-internal text (logs, exceptions, schema strings, SMS replies) is *not* wrapped. |

## 3. Schema change

A new table — not an ALTER of an existing one — because the current auth model uses a hardcoded `DEMO_USERS` dict in `app.py` rather than a `users` table. The new table is added through `database.py`'s existing SQLAlchemy `metadata` declaration, same as `interventions` (the Phase E table).

### New table `user_preferences`

```python
user_preferences = Table(
    "user_preferences", metadata,
    Column("username", Text, primary_key=True),
    Column("language", Text),  # nullable; NULL = "no preference set"
)
```

`username` is the existing identity primitive — it's what `session['username']` carries, what `illness_reports.submitter` records, and the keys of `DEMO_USERS`. No foreign key to `DEMO_USERS` (which isn't a table). If `DEMO_USERS` is later replaced by a real `users` table, a single ALTER adds the FK; until then, an orphan row in `user_preferences` (e.g., for a since-removed demo user) is harmless.

`init_db()` already calls `metadata.create_all(conn)` which creates the new table on first boot. No migration step needed.

**No CHECK constraint** on `language`. Two reasons:
1. SQLite can't `ALTER` a CHECK constraint in place; adding a 17th language later would force a table rebuild. The same friction applies on Postgres (the in-flight migration target per `MEMORY.md`'s sqlite_to_postgres_migration entry).
2. Validation already happens in `language.py`: `POST /lang` rejects unknown codes, and the locale-selector callback (§6) treats any out-of-allow-list value from the table as "not set" and falls through to the next layer.

The `LANGUAGES` dict in `language.py` is the single source of truth for allowed codes. Adding a language is one entry there plus the `make i18n-init LANG=<code>` translation files. No schema change ever needed.

## 4. Endpoint map

### New routes

| Method | Path | Auth | Purpose |
|---|---|---|---|
| POST | `/lang` | any (incl. anonymous) | Picker form submission. Validates `code`, sets cookie, writes `user_preferences.language` if signed in, redirects. |

### Changed routes

| Method | Path | Change |
|---|---|---|
| GET | `*` (every HTML response) | `_base.html` now wraps each page with the picker + the unverified-locale banner. |

### Unchanged

`POST /sms`, `POST /ingest`, `POST /actions`, `/login`, `/logout`, `/health`. These are programmatic or untranslated.

## 5. Architecture and file layout

```
App/backend/
├── language.py                      ← NEW: thin wrapper around Flask-Babel
├── babel.cfg                        ← NEW: tells pybabel which files to scan
├── translations/                    ← NEW: gettext catalog tree
│   ├── messages.pot                 ← generated; source-of-truth template
│   ├── en/LC_MESSAGES/messages.{po,mo}
│   ├── sn/LC_MESSAGES/messages.{po,mo}
│   └── nd/LC_MESSAGES/messages.{po,mo}
├── app.py                           ← wraps user-facing strings in _()
├── estimator.py                     ← wraps rationale strings in _()
├── database.py                      ← + user_preferences Table in metadata
├── requirements.txt                 ← + Flask-Babel
├── Makefile                         ← NEW or extended: i18n-extract/init/update/compile
└── templates/
    ├── _base.html                   ← NEW: shared layout w/ picker + unverified banner
    ├── _lang_picker.html            ← NEW: small partial for the dropdown
    └── (existing templates extend _base.html and wrap strings in {{ _() }})
```

### Module boundaries

- **`language.py` owns:**
  - Babel initialisation (`init_babel(app)`).
  - Allowed-language config: `LANGUAGES = {'en': 'English', 'sn': 'Shona', 'nd': 'isiNdebele'}`.
  - The locale-selector callback (§6).
  - The `POST /lang` view.
  - `current_lang()` / `set_lang(code, *, user_id=None)` helpers.
  - `unverified_locale(code)` — used by `_base.html` to decide whether to render the banner.
- **`app.py` knows:** "import `init_babel` from `language` and call it." Views just use `_('...')`. No locale logic in views.
- **`estimator.py` knows:** wrap each rationale string with `_()`. No other change.

### `_base.html` introduction

Today each template owns its own `<head>` and inline `<style>` (e.g. `login.html` is 134 lines, most of it CSS). Introducing `_base.html` for the picker + banner is also a chance to deduplicate that CSS into a shared head block. In-scope per the brainstorming skill's "improve code you're working in" guideline. Each existing template becomes `{% extends "_base.html" %}` with its body in a `{% block content %}`.

## 6. Locale-selector callback

Registered with Flask-Babel via `@babel.localeselector`. Runs on every request. Returns one of `'en' | 'sn' | 'nd'`. Precedence:

1. **URL override** — `?lang=sn` on the current request only. Used by `POST /lang`'s redirect so the picker change takes effect on the *next* render without waiting for a second roundtrip. Validated against `LANGUAGES.keys()`; invalid → ignored.
2. **`user_preferences.language`** — only when `session['username']` is present. Looked up by username. Validated; out-of-allow-list values are treated as NULL and fall through.
3. **Cookie** — `aqua_lang`, 1-year expiry, `SameSite=Lax`. Validated. Effectively the *anonymous* preference: only consulted when no user is signed in (since the DB column wins above), or as the fallback for signed-in users with `language=NULL`.
4. **`Accept-Language`** — `request.accept_languages.best_match(LANGUAGES.keys())`.
5. **Default** — `'en'`.

### Why DB above cookie

The target audience includes clinicians and government officials sharing workstations in clinics and offices. With DB-first, a Shona-speaking clinician signs in on any workstation and immediately sees Shona — their preference follows the account, not the machine. The next clinician signs in and gets *their* language. No manual picker clicks needed on shift change.

The cookie still serves a real purpose:
- **Anonymous users** (the login page is the obvious case): cookie wins because there's no DB row to consult.
- **Sign-out → sign-in cycle on the same device:** the cookie outlives the session, so the login page renders in the previous user's language until someone signs in.

### Trade-off accepted

A signed-in user *cannot* temporarily switch language on one device without it propagating across all their devices — every picker click writes the DB. This is a deliberate trade for the shared-workstation use case. Users who want a one-off switch can sign out (cookie then wins on that device) or use a different browser profile.

## 7. Picker UX

### Picker partial (`_lang_picker.html`)

Top-right of `_base.html`. A `<form method="POST" action="/lang">` with a `<select name="code">` containing the three options, auto-submitting on change via one inline event handler: `onchange="this.form.submit()"`. No dropdown library, no extra dependency. The selected option matches `current_lang()`.

### `POST /lang` route

- Form fields: `code` (required, must be in `LANGUAGES`).
- Side effects: sets the `aqua_lang` cookie; if `session['username']` is present, upserts a row in `user_preferences` (`INSERT … ON CONFLICT (username) DO UPDATE SET language = :code`).
- Validation failure (missing / unknown code): 400 with body "Invalid language code." (this string itself wrapped in `_()`).
- Response: `302` to `request.referrer` only if it starts with `/`, else `/dashboard`. Same referrer guard as `POST /actions`.

### Unverified-locale banner

Rendered by `_base.html` whenever `unverified_locale(current_lang())` returns True. Copy (wrapped in `_()` so it gets translated into the current language):

> *"Machine-translated draft — awaiting native review. Translations of medical terms may be inaccurate; for clinical decisions, switch to English."*

`unverified_locale(code)` returns:
- `False` for `'en'`.
- `False` for any language whose `.po` file contains *zero* `#, fuzzy` entries.
- `True` otherwise.

When a native speaker reviews `translations/sn/LC_MESSAGES/messages.po` and removes the `#, fuzzy` markers, the next `make i18n-compile` + app restart automatically drops the banner. No separate flag table to keep in sync.

## 8. Translation build workflow

Codified in `Makefile` targets to avoid drift:

```
make i18n-extract           pybabel extract -F babel.cfg -o translations/messages.pot .
make i18n-init LANG=sn      pybabel init -i translations/messages.pot -d translations -l sn
make i18n-update            pybabel update -i translations/messages.pot -d translations
make i18n-compile           pybabel compile -d translations --use-fuzzy
```

`--use-fuzzy` is intentional: without it, machine-translated entries don't compile to `.mo` and the Shona/Ndebele UIs would silently fall back to English — defeating the entire demo. The `unverified_locale()` banner is what surfaces the "unverified" status honestly to users.

### `babel.cfg`

Three extractor entries:
- `[python: **.py]` with `keywords=_,gettext,ngettext` and `ignore=tests/,__pycache__/,WNTR/,.venv/`.
- `[jinja2: templates/**.html]` with `extensions=jinja2.ext.i18n`.

### Machine-translation procedure

One-time per language addition, outside the request path. Implemented as a short script in `scripts/translate_po.py`:

1. After `pybabel extract`, parse `messages.pot` → list of `msgid` strings.
2. Use the `googletrans` Python library (no API key, no quota for the volumes we have).
3. Write the result back into `translations/<lang>/LC_MESSAGES/messages.po` with every entry marked `#, fuzzy`.
4. Commit the `.po` *and* the compiled `.mo`.

### `.mo` files in git

Tracked. At 16 languages × ~50 KB each it's <1 MB — trivial. Railway deploy doesn't need `pybabel` installed; the compiled `.mo` rides along in the repo. If this ever grows uncomfortable, move compilation to CI.

## 9. What gets wrapped in `_()`

| File / area | Wrap? | Notes |
|---|---|---|
| All six templates (`login`, `dashboard`, `medical_report`, `medical_history`, `medical_report_detail`, `dashboard_report_detail`) | Yes | Every visible English literal. Brand strings ("Aqua Solutions", product name "Water-safety pipeline") deliberately left **un**wrapped — brand names don't translate. |
| The "NOT a cholera detector" disclaimer | Yes, carefully | This is the project's main safety claim. The English source string is the authoritative version; translators must preserve the negation. Flagged in §13 risks. |
| `app.py` flash messages, error strings, page titles | Yes | Including validation errors from `POST /lang`, `POST /actions`, `POST /login`. |
| `estimator.py` rationale strings | Yes | All four (LOW / MEDIUM / HIGH / SEVERE) render in the gov detail page. |
| SMS reply strings in `app.py` (the `POST /sms` state-machine outputs from Phase F) | **No** | SMS leg is out of scope for v1 (Section 2). |
| `database.py`, `engine.py`, `sensor_ingest.py`, `feature_engineering.py` | **No** | Server-internal: logs, exceptions, schema strings. Wrapping them would translate stack traces in some Python locales and adds no user benefit. |
| Estimator rule names ("close_borehole", "dispatch_sample_team") | **No** | Programmatic identifiers, never shown raw to users. The human-readable label on the button *is* wrapped (separately in the template). |

### Plurals

Case-count strings ("1 case" vs "5 cases") use `ngettext('%(n)d case', '%(n)d cases', n)`. Identified locations:
- `dashboard.html` recent-reports row.
- `medical_history.html` reports-table row.
- `dashboard_report_detail.html` and `medical_report_detail.html` metadata block.

Three places. Hand-wrap each; don't retrofit a plural framework elsewhere.

## 10. Testing plan

New module `App/backend/tests/test_i18n.py` using the existing `conftest.py` scratch-DB fixtures. Cases:

1. **Locale selector precedence.** Set up a signed-in user with a `user_preferences` row (`username='dr.smith', language='sn'`) inserted via a fixture. Hit `/dashboard` with cookie `aqua_lang=nd`, then with `?lang=en` on top of that. Each layer wins in order: URL > DB > cookie > Accept-Language > 'en'. Additionally: hit `/dashboard` *anonymously* with cookie `aqua_lang=nd` and confirm the cookie wins (since no DB row applies).
2. **Picker round-trip.** `POST /lang code=sn` from `/dashboard` redirects back to `/dashboard`, sets the cookie, upserts `user_preferences (username='dr.smith', language='sn')`. `POST /lang code=zz` returns 400.
3. **Referrer guard.** `POST /lang` with `Referer: https://evil.example/x` ignores the referrer and redirects to `/dashboard`.
4. **Estimator wrapping.** `estimate_risk_tier()` called under `with app.test_request_context()` and a forced `sn` locale returns the Shona rationale string (assert it is not the English literal). Compile a minimal fixture `.mo` for the test.
5. **Fallback.** Force locale `'sn'` against a deliberately-empty `sn` catalog and verify the response body contains English (not the bare `msgid`, not a 500).
6. **Banner gating.** `unverified_locale('en')` is False; `unverified_locale('sn')` is True while fuzzy entries exist; after a fixture removes the fuzzy markers from the in-memory catalog, the helper flips to False.

### Manual smoke before commit

Sign in, change picker to Shona, click through every page in the nav, confirm no `[Missing: ...]` markers and that the banner shows everywhere. Repeat for Ndebele. Sign out and confirm the picker still works on `/login`.

## 11. Deploy notes

- Railway: no env-var changes. `Flask-Babel` ships in `requirements.txt`; compiled `.mo` files are in the repo.
- `make i18n-compile` runs as part of the developer's pre-commit ritual when `.po` files change. A one-line note at the bottom of `CLAUDE.md` documents this so future Claude sessions know to recompile after editing translations.
- No migration script needed — the new `user_preferences` table is created on next boot by the existing `metadata.create_all(conn)` call in `init_db()`. Same pattern as the Phase E `interventions` table.
- The `googletrans` dependency lives in `requirements-dev.txt` (new file) or is run via `pipx`, *not* in production `requirements.txt` — translation happens offline.

## 12. Out of scope (intentional)

- **SMS leg.** Inbound symptom-keyword parsing and outbound replies stay English in v1. Earns its own design when needed.
- **The other 13 official languages.** The framework supports them; translations are not produced yet. Adding one is `make i18n-init LANG=<code>` + translate + add a single entry to the `LANGUAGES` dict in `language.py`. No schema change.
- **Right-to-left handling.** None of Zimbabwe's 16 official languages are RTL.
- **Locale-aware number, date, currency formatting** beyond the existing `DD/MM` convention. Babel offers this; not in v1.
- **Language-aware ML behavior.** Server-internal text (sensor-reading column names, model card text) stays English.
- **Admin UI for managing translations.** Translators edit `.po` files via a PR.
- **Native-speaker review workflow itself.** Out-of-band; the system surfaces the unverified banner but doesn't manage the review queue.
- **Language detection from the SMS sender's phone number / country code.** Not implemented.
- **Translating the SMS dialog state-machine prompts in `app.py`'s `/sms` handler** — explicitly part of "SMS leg" above.

## 13. Risks

- **Machine-translated medical terms may be wrong.** Google Translate's Shona/Ndebele rendering of "diarrhoea", "dehydration", "cholera", or "borehole" may not match the clinical vocabulary actually used in Zimbabwean health practice. *Mitigation:* the unverified banner is honest about this and explicitly redirects clinical users to English. Native review is the only real fix.
- **The "NOT a cholera detector" disclaimer translation must preserve the negation.** A mistranslation here would invert the project's central safety claim. *Mitigation:* this string is flagged in the `.po` files with a `#. TRANSLATORS:` comment explaining the stakes; native review of the disclaimer is the highest-priority verification item.
- **`Accept-Language` is unreliable in Zimbabwe.** Phones often ship configured to English even when the user prefers Shona. *Mitigation:* picker ranks above `Accept-Language` by design. Worth checking in the first user-testing session with Allen Chafa.
- **DB-over-cookie precedence means no per-device temporary switch.** A signed-in clinician can't switch one device to English to check a Shona translation without it propagating to all their devices. *Mitigation:* sign out (cookie then wins) or use a different browser profile for the temporary view. Documented in §6.
- **`.mo` files as binary artefacts in git.** At 16 languages × ~50 KB this is trivial today. Flagged as something to revisit if the catalog grows substantially.
- **`googletrans` is an unofficial wrapper around Google Translate's web endpoint.** It can break without notice. *Mitigation:* we only invoke it offline during translation generation; if it breaks, paste-into-the-website fallback works. No runtime dependency.

## 14. Implementation phasing

Suggested order, each phase committable on its own:

1. **Plumbing.** Add `Flask-Babel` to `requirements.txt`, create `language.py`, `babel.cfg`, the `Makefile` targets, the `user_preferences.language` migration, the empty `translations/` tree with the English catalog populated. App still renders 100% English. Verify no regression.
2. **Template refactor.** Introduce `_base.html` + `_lang_picker.html`. Convert the six existing templates to `{% extends "_base.html" %}`. Wrap every visible string in `{{ _() }}`. Still English-only; confirm no visual regression.
3. **Picker + selector + locale logic.** Wire up the locale-selector callback, `POST /lang`, the cookie, the DB write. Picker is functional but switching to Shona/Ndebele still renders English (no catalogs yet). Add the test suite.
4. **Shona + Ndebele machine translation.** Run `scripts/translate_po.py`, commit the `.po` and `.mo`. The unverified banner activates. Smoke-test all pages in both languages.
5. **`estimator.py` rationale strings + plural retrofits.** Three plural sites. Recompile.

Phases 1–3 are zero-translation: the framework is fully in place and tested before any non-English text exists. Phases 4–5 turn the translations on.
